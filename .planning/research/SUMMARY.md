# Project Research Summary

**Project:** LitUI v10.1 Component Knowledge Image
**Domain:** Node.js programmatic XML compilation + text-to-image generation for AI knowledge base
**Researched:** 2026-03-01
**Confidence:** HIGH

## Executive Summary

LitUI v10.1 is a maintainer-only tooling milestone that adds two generated artifacts to the existing v10.0 monorepo: a structured XML compilation of all 34 component skill files (`skill/lit-ui-knowledge.xml`) and a condensed PNG reference image rendered from that XML (`skill/lit-ui-knowledge.png`). Both artifacts are generated by standalone TypeScript scripts in `scripts/`, committed to git, and automatically propagated to consumers via the existing `injectOverviewSkills()` CLI pipeline — no changes to existing packages or consumers are required. The scope is deliberately narrow: two new scripts, two new generated files, three new root `package.json` entries.

The recommended approach uses `@napi-rs/canvas` (Skia-backed, pre-built binaries, zero system dependencies) for image rendering — explicitly not `node-canvas` (Cairo-backed, requires system libraries, has known `registerFont()` bugs on Linux) and explicitly not Puppeteer/headless browsers (banned, 300MB+). The rendering pipeline is a standard two-pass pattern: first measure all text to compute total canvas height, then create the canvas at that exact height and render. The XML compiler is a pure Node.js `fs.readdir` walk with XML escaping — no third-party parsing libraries needed. Both scripts run via `node --experimental-strip-types` (available in the installed Node.js v25.3.0) without a separate compilation step.

The dominant risk class for this milestone is font registration failure. `GlobalFonts.registerFromPath()` fails silently in multiple real-world scenarios (wrong path, wrong family name alias, wrong call order, unsupported font format) and produces output that looks correct but is rendered in the wrong font — breaking all line-width calculations based on monospace character metrics. This risk is fully preventable: bundle a `.ttf` font file (JetBrains Mono, OFL license), use absolute paths via `fileURLToPath(import.meta.url)`, call `GlobalFonts.registerFromPath()` before any canvas creation, and immediately validate the registration with `ctx.measureText('i').width === ctx.measureText('W').width` (monospace invariant check).

## Key Findings

### Recommended Stack

The stack is additive to the working v10.0 monorepo. One new production dependency is required: `@napi-rs/canvas@^0.1.95` added as a root dev-only dependency (or scoped to the scripts workspace). It ships pre-built Skia binaries for all target platforms (macOS arm64/x64, Linux x64 gnu/musl, Windows x64) — no `node-gyp` compile step, no system Cairo/Pango installation. `fast-xml-parser` is optional and only needed for structured section-level XML extraction; for verbatim text rendering it is unnecessary. No new workspace packages, no new build tooling.

**Core technologies:**
- `@napi-rs/canvas@^0.1.95`: Canvas 2D API in Node.js for text rendering — chosen over `node-canvas` (Cairo system dependency, Linux `registerFont()` bugs #2255/#2285/#2097) and Puppeteer (banned, 300MB+); pre-built Skia binaries for all CI/dev targets
- Node.js built-in `fs/promises` + `path`: XML compiler has zero new dependencies — pure string operations
- `fast-xml-parser@^4.5.1` (optional): XML parsing for structured section extraction — pure JS, zero dependencies, 6M+ weekly downloads; skip entirely for verbatim text rendering
- `node --experimental-strip-types`: Runs TypeScript scripts directly in Node.js v25.3.0 without `tsx` or a compile step

**Critical version constraint:** Use `GlobalFonts.registerFromPath(pathString, familyName)` only — NOT `GlobalFonts.register(buffer)`. The buffer overload has a confirmed memory aliasing bug (issue #1006, Feb 2025) where Buffer GC silently invalidates the registered font after garbage collection.

**tsup note:** If the renderer is ever compiled through tsup, `@napi-rs/canvas` must be listed in `external` in `tsup.config.ts` — native binary packages cannot be bundled by esbuild.

### Expected Features

The milestone has exactly two deliverables, both required for launch. There is no partial MVP — the XML artifact and PNG artifact are the milestone.

**Must have (table stakes):**
- XML compilation script — reads all 34 skill files, strips YAML frontmatter, wraps each in `<skill name="...">`, XML-escapes content (`<`, `>`, `&` in TypeScript generics and Markdown code blocks), writes `skill/lit-ui-knowledge.xml`
- PNG renderer — reads compiled XML as plain text (no XML parsing in renderer), normalizes whitespace, two-pass height measurement, renders at 8pt monospace on white background with 1500px width, writes `skill/lit-ui-knowledge.png`
- Bundled JetBrains Mono TTF font (OFL license) — committed to `scripts/fonts/` or `packages/cli/src/fonts/`; ensures consistent output across macOS, Linux CI, and Windows
- Standalone runnable scripts — executable as `node --experimental-strip-types scripts/compile-knowledge.ts` with no prior build step; wired as `knowledge:compile`, `knowledge:render`, `knowledge:build` in root `package.json`
- XML entity decoding before rendering — `&amp;` → `&`, `&lt;` → `<`, `&gt;` → `>` must decode in the renderer so display characters appear in the PNG, not literal entity sequences

**Should have (add after validation):**
- Section-level XML `<section title="...">` sub-elements — enables structured AI queries by component section (Props, Behavior Notes, CSS Custom Properties, etc.)
- Component separator lines in PNG — thin horizontal rule + component name label between skill sections for visual navigation
- CI integration — `pnpm knowledge:build` as a post-build step to keep artifacts current

**Defer (v2+):**
- Incremental re-generation (mtime-based, only re-render changed sections)
- Multi-page PDF output for paginated reference consumers

**Anti-features (explicitly do not build):**
- Syntax highlighting in PNG — adds a tokenizer dependency (e.g., shiki), marginal benefit at 8pt, renders all text in single color instead
- PDF output — wrong format for AI image context tools; XML file serves machine-searchable purpose
- Automatic file-watcher re-generation — runtime complexity inappropriate for a build-time artifact
- SVG intermediate + sharp — sharp/librsvg cannot reliably embed custom fonts in CI environments without system font installation

### Architecture Approach

The pipeline is two sequential scripts connected by a filesystem artifact. The compiler (`scripts/compile-knowledge.ts`) walks `skill/` with `fs.readdir({ recursive: true })`, collects all `SKILL.md` files, sorts them for deterministic output, strips YAML frontmatter, XML-escapes content, and writes `skill/lit-ui-knowledge.xml`. The renderer (`scripts/render-knowledge-image.ts`) reads the XML as plain text (no XML parsing — just `split('\n')` after entity decoding), runs two-pass height measurement using a probe canvas, creates the final canvas at computed height, and writes `skill/lit-ui-knowledge.png`. Both scripts are standalone in `scripts/`, not part of any published package. Generated artifacts are committed to git and automatically included in `injectOverviewSkills()` copies and CLI npm publishes without any pipeline changes.

**Major components:**
1. `scripts/compile-knowledge.ts` — Skill file aggregator: recursive `fs.readdir` walk, frontmatter strip, XML escape, structured XML output to `skill/lit-ui-knowledge.xml`
2. `skill/lit-ui-knowledge.xml` — Intermediate artifact (git-committed): ~225KB structured knowledge document, independently useful for programmatic querying by AI tools
3. `scripts/render-knowledge-image.ts` — Canvas renderer: entity decode, two-pass text layout (probe canvas + final canvas), `@napi-rs/canvas` 2D API, 8pt monospace, white background, PNG output
4. `skill/lit-ui-knowledge.png` — Final output (git-committed): 1500px wide, dynamically tall (~3,500 lines, ~42,000px estimated), condensed monospace reference image
5. Root `package.json` — Three new script entries: `knowledge:compile`, `knowledge:render`, `knowledge:build`

**Key architectural patterns:**
- Recursive `fs.readdir` walk (not hardcoded file list) — zero maintenance as new components are added in future milestones; new `SKILL.md` files appear in XML automatically
- Two-pass render (probe canvas to measure → compute height → create final canvas → render) — mandatory; node-canvas/skia-canvas height is fixed at creation time, cannot change after drawing
- Scripts-only (not CLI commands) — `canvas` is a root dev-only dependency; shipping it inside `packages/cli` would impose Cairo as a system dependency on every `npx lit-ui` consumer, breaking zero-system-dependency DX
- Commit generated artifacts — `skill/lit-ui-knowledge.xml` and `skill/lit-ui-knowledge.png` are committed (not gitignored) so `injectOverviewSkills()` includes them without a mandatory pre-install build step

### Critical Pitfalls

All critical and high-severity pitfalls concentrate in Phase 1 (canvas setup and font registration). Once font registration and two-pass layout are validated on a 50-line sample, the remaining implementation is low-risk.

1. **Font silently falls back to "Sans" — no exception thrown** (CRITICAL-01/CRITICAL-04) — `GlobalFonts.registerFromPath()` appears to succeed but text renders in the system fallback font, breaking all line-width calculations. Prevention: use absolute path via `fileURLToPath(import.meta.url)`, use the exact registered family string in `ctx.font`, call registration before any canvas creation, validate with `ctx.measureText('i').width === ctx.measureText('W').width`.

2. **Fixed canvas height cuts off content or wastes space** (CRITICAL-02) — Canvas height is fixed at creation time; resizing after drawing clears all pixel data. Prevention: strict two-pass architecture — measure all lines in pass 1 using a `createCanvas(1500, 1)` probe canvas, compute `totalHeight = lineCount * lineHeight + padding`, create final canvas at that exact height in pass 2.

3. **XML entity sequences render literally in PNG** (CRITICAL-03) — The knowledge XML contains `&amp;`, `&lt;`, `&gt;` from the compiler's escape step. If not decoded before `fillText()`, these render as literal text. Prevention: decode entities as the first step when reading the XML file for rendering, before splitting into lines.

4. **`measureText()` diverges from browser Canvas by 2–4px per character** (HIGH-01) — Causes line overflow past the 1500px right edge with no error. Prevention: for monospace fonts, measure one reference character once (`const charWidth = ctx.measureText('M').width`) and use character-count-based wrapping throughout — never per-word `measureText()` for layout decisions.

5. **Native dependency build failure in CI** (HIGH-02) — `node-canvas` (Cairo) requires system libraries absent in minimal CI/Docker environments. Full mitigation: use `@napi-rs/canvas` (Skia, pre-built binaries, no system dependencies) — this eliminates HIGH-02 entirely. Note: ARCHITECTURE.md references `node-canvas` in code examples; the STACK.md recommendation of `@napi-rs/canvas` supersedes this for implementation.

## Implications for Roadmap

This milestone maps to a single focused implementation window with a strict internal build order. The dependency graph forces the sequence: font/canvas validation first, then compiler (no external deps), then renderer (depends on both validated canvas and XML artifact), then wiring and commit. There is no logical multi-phase split — the milestone is self-contained and estimated at 2–4 hours of implementation.

### Phase 1: Foundation — Font Setup and Canvas Validation
**Rationale:** All critical pitfalls (CRITICAL-01 through CRITICAL-04, HIGH-01 through HIGH-05) are encountered in canvas and font setup. This must be solved and validated on a short sample before any content-generation code is written — a wrong font or wrong canvas height assumption silently corrupts all downstream work.
**Delivers:** Validated `@napi-rs/canvas` install, bundled JetBrains Mono TTF registration with `GlobalFonts.registerFromPath()`, monospace invariant check passing (`measureText('i').width === measureText('W').width`), two-pass layout skeleton running on a 50-line sample, anti-aliasing mode set (`ctx.antialias = 'default'`, `ctx.textBaseline = 'top'`, integer Y coordinates)
**Addresses:** CRITICAL-01/CRITICAL-04 (font registration), HIGH-01 (character-count-based wrapping), HIGH-05 (anti-aliasing at 8pt)
**Avoids:** Writing layout logic before font metrics are verified; using `GlobalFonts.register(buffer)` (known bug #1006)

### Phase 2: XML Compiler
**Rationale:** No external dependencies required (pure Node.js `fs/promises`). Independent of canvas — can be written and validated before touching image rendering. Produces the intermediate artifact that the renderer consumes and that is useful independently for AI tools.
**Delivers:** `scripts/compile-knowledge.ts` producing `skill/lit-ui-knowledge.xml` — all 34 skill files, YAML frontmatter stripped, content XML-escaped (`&`, `<`, `>` → `&amp;`, `&lt;`, `&gt;`), router skill ordered first, sub-skills sorted alphabetically for deterministic output
**Implements:** Walk-and-compile architecture pattern with `fs.readdir({ recursive: true })`
**Avoids:** Anti-Pattern 1 (hardcoded file list), Anti-Pattern 6 (rendering without XML escaping), CRITICAL-03 (unescaped XML characters causing malformed XML)

### Phase 3: PNG Renderer
**Rationale:** Depends on Phase 1 (validated canvas/font setup) and Phase 2 (XML artifact to read). Implements the full two-pass render using canonical line count from the XML file.
**Delivers:** `scripts/render-knowledge-image.ts` producing `skill/lit-ui-knowledge.png` — entity-decoded XML as input, two-pass height measurement, 1500px × dynamically tall canvas, 8pt JetBrains Mono, white background (`#ffffff`), black text (`#000000`)
**Uses:** `@napi-rs/canvas`, bundled JetBrains Mono TTF, character-count-based line wrapping via single `measureText('M')` reference measurement
**Implements:** Fixed-width canvas text rendering pattern
**Avoids:** CRITICAL-02 (fixed canvas height), CRITICAL-03 (entity sequences in output), HIGH-02 (CI native dependency failure — eliminated by using @napi-rs/canvas), HIGH-03 (memory exhaustion — use `createPNGStream()` if canvas height exceeds ~50,000px), HIGH-04 (async toBuffer memory leak — use sync form or stream)

### Phase 4: Wiring, Artifacts, and Sync to CLI
**Rationale:** Final integration — wire the two scripts, run the full pipeline on all 34 skill files, commit both generated artifacts, and sync to `packages/cli/skill/` for the next CLI publish.
**Delivers:** Three new root `package.json` script entries (`knowledge:compile`, `knowledge:render`, `knowledge:build`); committed `skill/lit-ui-knowledge.xml` and `skill/lit-ui-knowledge.png`; `packages/cli/skill/` updated with both new artifacts; CONTRIBUTING.md note that `pnpm knowledge:build` must be run after modifying any skill file
**Addresses:** Anti-Pattern 4 (not committing generated artifacts); integration with existing CLI publish pipeline (`packages/cli/package.json` `"files": ["dist", "skill"]` already covers new artifacts)

### Phase Ordering Rationale

- Canvas/font validation comes first because all layout code depends on correct font metrics — invalid metrics corrupt everything downstream silently with no error thrown
- XML compiler comes before the renderer because it is dependency-free, validates the content pipeline independently, and produces the renderer's input; early validation of XML escaping catches issues before rendering
- Renderer comes third because it depends on both validated canvas setup (Phase 1) and XML content (Phase 2)
- Wiring and artifact commit comes last because it requires the full pipeline to be verified end-to-end before committing the generated files to git

### Research Flags

All phases have standard, well-documented patterns. No phase requires a dedicated research spike before implementation.

Phases with standard patterns (skip research-phase):
- **Phase 1 (Font Setup):** `GlobalFonts.registerFromPath()` API verified from @napi-rs/canvas official GitHub with code examples. Font registration ordering constraint documented in node-canvas issues #2285/#2097. Monospace invariant test is a trivial two-line verification.
- **Phase 2 (XML Compiler):** `fs.readdir({ recursive: true })` is a Node.js built-in with official docs. XML escaping is a standard string replacement. YAML frontmatter stripping is a regex boundary detection.
- **Phase 3 (PNG Renderer):** Two-pass canvas render is a documented pattern verified from node-canvas GitHub and multiple community tutorials. Character-count-based wrapping for monospace is a ~10-line algorithm.
- **Phase 4 (Wiring):** Root `package.json` scripts with `node --experimental-strip-types` follow the established `scripts/install-skill.mjs` precedent exactly.

Potential empirical validation needed (not research — just a quick test):
