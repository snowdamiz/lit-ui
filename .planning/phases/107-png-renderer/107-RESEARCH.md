# Phase 107: PNG Renderer - Research

**Researched:** 2026-03-02
**Domain:** @napi-rs/canvas, XML parsing in Node.js, two-pass PNG layout, HTML entity decoding
**Confidence:** HIGH

## Summary

Phase 107 builds `scripts/render-knowledge-image.ts` — a single TypeScript script that reads the compiled `skill/lit-ui-knowledge.xml` produced by Phase 106, extracts all skill/section content, decodes XML entities back to display characters, and renders the full text as a condensed monospace PNG at exactly 1500px wide. The canvas/font foundation (Phase 105) is already validated: `@napi-rs/canvas` is installed, `JetBrainsMono-Regular.ttf` is bundled in `scripts/fonts/`, font registration with `GlobalFonts.registerFromPath()` works, and the two-pass layout pattern (probe canvas at 1500x1 → compute height → final canvas at exact height) is confirmed working.

The XML file is 3984 lines covering 33 skills. The renderer must: (1) parse the XML to extract skill names and their section content, (2) decode `&amp;`, `&lt;`, `&gt;`, `&quot;` entities so TypeScript code renders as `<T>` not `&lt;T&gt;`, (3) layout all text using the two-pass pattern with a thin horizontal rule and skill label between skills, and (4) write the final PNG to `skill/lit-ui-knowledge.png`. All technical foundations are de-risked — this phase is primarily about implementing the content rendering logic on top of the validated skeleton.

For XML parsing, Node.js's built-in `node:readline` + manual regex-based extraction or the `xmldom` / `fast-xml-parser` npm packages are the choices. Given the project already uses `node --experimental-strip-types` without compilation and the XML structure is simple and well-known (generated by the project itself), a lightweight hand-rolled parser using regex against the raw XML string is sufficient and avoids new dependencies. However, because the XML entity encoding is deterministic (all content produced by `xmlEscape()` in compile-knowledge.ts), a pure regex approach over the raw file content is HIGH confidence for this specific use case.

**Primary recommendation:** Read XML as a string, extract skill names via `<skill name="...">` regex, extract section content via `<section title="...">...</section>` regex, decode entities with a 4-replacement function, then render text line-by-line using the validated two-pass canvas pattern with `ctx.strokeRect`-based separator lines between skills.

### Pattern 3: XML Skill Extraction via Regex
**What:** Extract skill name + all section content from the generated XML using known structure.
**When to use:** After reading the XML file as a UTF-8 string.

The XML structure is always:
```xml
<skill name="SkillName">
  <section title="SectionTitle">
content lines here
  </section>
</skill>
```

```typescript
// Source: Derived from compile-knowledge.ts structure (skill/lit-ui-knowledge.xml is
// machine-generated by this project — structure is 100% deterministic)
interface SkillData {
  name: string
  content: string[]  // all text lines, pre-decoded
}

function parseSkills(xml: string): SkillData[] {
  const skills: SkillData[] = []

  // Match each <skill name="...">...</skill> block
  const skillRegex = /<skill name="([^"]*)">([\s\S]*?)<\/skill>/g
  let skillMatch: RegExpExecArray | null

  while ((skillMatch = skillRegex.exec(xml)) !== null) {
    const name = decodeEntities(skillMatch[1])
    const skillBody = skillMatch[2]

    // Extract all lines from all sections within this skill
    const lines: string[] = []
    const sectionRegex = /<section title="([^"]*)">([\s\S]*?)<\/section>/g
    let sectionMatch: RegExpExecArray | null

    while ((sectionMatch = sectionRegex.exec(skillBody)) !== null) {
      const sectionTitle = decodeEntities(sectionMatch[1])
      const sectionContent = decodeEntities(sectionMatch[2])

      // Add section title as a subheading, then content lines
      lines.push(`### ${sectionTitle}`)
      const contentLines = sectionContent.split('\n')
      for (const line of contentLines) {
        lines.push(line)
      }
    }

    skills.push({ name, content: lines })
  }

  return skills
}
```

### Pattern 4: Entity Decoding
**What:** Reverse the 4 substitutions made by `xmlEscape()` in compile-knowledge.ts.
**When to use:** On all extracted text BEFORE passing to `ctx.fillText()`.

The `xmlEscape()` function in compile-knowledge.ts applies in this order:
1. `&` → `&amp;`
2. `<` → `&lt;`
3. `>` → `&gt;`
4. `"` → `&quot;`

Decoding must reverse in opposite order (or any order is safe since none of the decoded chars are themselves entity-start `&`):
```typescript
// Source: Derived from compile-knowledge.ts xmlEscape() — exact inverse
function decodeEntities(text: string): string {
  return text
    .replace(/&quot;/g, '"')
    .replace(/&gt;/g, '>')
    .replace(/&lt;/g, '<')
    .replace(/&amp;/g, '&')   // MUST be last — prevents double-decoding
}
```

**Critical:** `&amp;` must be decoded LAST. If decoded first, `&amp;lt;` would become `&lt;` (a raw entity) rather than the correct `<`. The original encoder put `&` first; the decoder must put `&amp;` last.

### Pattern 5: Two-Pass Rendering with Skill Separators
**What:** Two-pass layout — probe pass measures total height, render pass writes pixels. Between each skill block, draw a thin horizontal rule and skill name label.
**When to use:** Main rendering function.

```typescript
// Source: scripts/validate-canvas.ts two-pass skeleton (Phase 105 — confirmed working)
// Extended with skill separator logic
const CANVAS_WIDTH = 1500
const FONT_SIZE = 8
const LINE_HEIGHT = FONT_SIZE * 1.5      // 12px
const PADDING = 20
const SEPARATOR_HEIGHT = LINE_HEIGHT * 2  // space for rule + label

function measureAndRender(
  ctx: CanvasRenderingContext2D,
  skills: SkillData[],
  render: boolean
): number {
  ctx.font = `${FONT_SIZE}px JetBrainsMono`
  let y = PADDING

  for (let i = 0; i < skills.length; i++) {
    const skill = skills[i]

    // Separator before each skill (except the first)
    if (i > 0) {
      if (render) {
        // Thin horizontal rule
        ctx.strokeStyle = '#000000'
        ctx.lineWidth = 0.5
        ctx.beginPath()
        ctx.moveTo(PADDING, y + LINE_HEIGHT / 2)
        ctx.lineTo(CANVAS_WIDTH - PADDING, y + LINE_HEIGHT / 2)
        ctx.stroke()

        // Skill name label on next line
        ctx.fillStyle = '#000000'
        ctx.fillText(`── ${skill.name} ──`, PADDING, y + LINE_HEIGHT * 1.5)
      }
      y += SEPARATOR_HEIGHT
    } else if (render) {
      // First skill: just the label at top
      ctx.fillStyle = '#000000'
      ctx.fillText(`── ${skill.name} ──`, PADDING, y)
    }
    y += LINE_HEIGHT

    // Content lines
    for (const line of skill.content) {
      if (render) {
        ctx.fillStyle = '#000000'
        ctx.fillText(line, PADDING, y)
      }
      y += LINE_HEIGHT
    }
  }

  return y + PADDING  // total height
}

// Pass 1: probe canvas — measure height only
const probeCanvas = createCanvas(CANVAS_WIDTH, 1)
const probeCtx = probeCanvas.getContext('2d')
const computedHeight = measureAndRender(probeCtx, skills, false)

// Pass 2: final canvas at exact height
const finalCanvas = createCanvas(CANVAS_WIDTH, computedHeight)
const finalCtx = finalCanvas.getContext('2d')

// White background
finalCtx.fillStyle = '#ffffff'
finalCtx.fillRect(0, 0, CANVAS_WIDTH, computedHeight)

// Render all content
measureAndRender(finalCtx, skills, true)

// Encode and write
const pngBuffer = await finalCanvas.encode('png')
await fs.writeFile(PNG_PATH, pngBuffer)
console.log(`PNG written: ${CANVAS_WIDTH}x${computedHeight}px → ${PNG_PATH}`)
```

### Pattern 6: Estimating Output Size
**What:** 3984-line XML → ~3000 rendered text lines → estimated PNG height.
**Calculation:**
- 3984 XML lines; each skill has structural tags, each content line maps roughly 1:1 to rendered lines
- 33 skills × ~110 lines average = ~3600 text lines
- 3600 × 12px LINE_HEIGHT + 33 separators × 24px + 2 × 20px padding ≈ **43,992px** tall
- At 1500px wide, 8pt JetBrains Mono, PNG file size estimate: ~1-3 MB (PNG compression very effective on text)

This is well within `@napi-rs/canvas` capabilities — Skia handles large canvases without issues.

### Anti-Patterns to Avoid
- **Decoding `&amp;` before other entities:** Will cause double-decode (`&amp;lt;` → `&lt;` instead of `<`). Always decode `&amp;` last.
- **Creating final canvas before probe pass:** Height is unknown until all lines are counted. Guessing causes truncation or padding. Always run the probe pass first.
- **Using `ctx.font` without setting it on both probe and final contexts:** Font metrics must be identical between passes for accurate height measurement. Set `ctx.font = '8px JetBrainsMono'` on every canvas context before measuring.
- **Using `GlobalFonts.register(buffer)` instead of `registerFromPath()`:** Memory aliasing bug (#1006, Feb 2025). The project STATE.md locks this decision — use `registerFromPath()` only.
- **Splitting XML content lines naively on `\n\n`:** Section content can contain blank lines (markdown blank lines between list items, code blocks with spacing). Split on `\n` and preserve blank lines as empty renders — they are intentional whitespace.
- **Calling `ctx.fillText()` with undefined or null:** If `decodeEntities` receives an undefined value (e.g., a regex match group that didn't capture), `fillText(undefined, ...)` renders the string "undefined". Add null-guard or use `?? ''`.
- **Not trimming leading/trailing whitespace from section content:** The XML `<section>` content has 2-space indentation from the compiler (the `el.split('\n').map((l) => '  ' + l)` join in compile-knowledge.ts). Each extracted content line will begin with "  " from the outer indent. Strip with `.trim()` or be aware the extra indent means content starts 2 characters right of the margin — either is fine but choose consistently.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| XML parsing | Full DOM parser or SAX streaming | Regex over known structure | The XML is machine-generated by this project; structure is 100% deterministic; a 5-line regex is sufficient and zero-dep |
| Entity decoding | Full HTML entity decoder | 4-replacement `.replace()` chain | The XML encoder uses exactly 4 substitutions; any other entity sequence can't appear in this file |
| PNG file writing | Manual pixel buffer manipulation | `canvas.encode('png')` + `fs.writeFile` | Built into `@napi-rs/canvas`; async Skia compression in libuv thread pool |
| Line wrapping | Word-wrap algorithm | None needed | Content is pre-formatted markdown; renders as-is; no wrapping required at 1500px (long lines may overflow but that's acceptable for a reference image) |
| Font metrics | Custom glyph measurement | `ctx.measureText()` | Canvas 2D standard API; monospace font means all characters are `charWidth` px wide; no need to measure individual lines |

**Key insight:** Because both the XML generation (Phase 106) and consumption (Phase 107) are owned by this project, the renderer can safely rely on the exact structure the compiler produces, avoiding the complexity of a general-purpose XML parser.

## Common Pitfalls

### Pitfall 1: Entity Decode Order (Double-Decode Bug)
**What goes wrong:** TypeScript generics like `<T>` appear as `&lt;T&gt;` in the output PNG instead of `<T>`.
**Why it happens:** If `&amp;` is decoded before `&lt;`/`&gt;`, the sequence `&amp;lt;` becomes `&lt;` (a raw entity string) rather than `<`. Subsequent passes don't run, so `&lt;` stays literal in the output.
**How to avoid:** Always decode `&amp;` LAST in the chain. The order must be: `&quot;` → `&gt;` → `&lt;` → `&amp;`.
**Warning signs:** PNG shows `&lt;T&gt;` or `&amp;` literally in TypeScript code examples.

### Pitfall 2: Probe Canvas Font Not Set
**What goes wrong:** Probe pass measures height incorrectly; final canvas is wrong size (either truncated or has massive whitespace at bottom).
**Why it happens:** `ctx.measureText()` and `ctx.fillText()` use the canvas's current font. If the probe context's `font` is not set, it defaults to `'10px sans-serif'`, producing different line metrics than the final `'8px JetBrainsMono'`.
**How to avoid:** Set `ctx.font = '8px JetBrainsMono'` on BOTH the probe context and the final context before measuring/rendering.
**Warning signs:** PNG height wildly different from expected; text overflows the bottom of the image, or large empty area at bottom.

### Pitfall 3: Line Height Mismatch Between Passes
**What goes wrong:** Final canvas height is correct but text is cut off at the bottom, or the last skill's content is missing.
**Why it happens:** The probe pass and render pass use a shared `measureAndRender` function that must produce the same `y` accumulator value for both `render=false` and `render=true` modes. Any conditional logic that takes different code paths between modes (e.g., rendering a separator only when `render=true` but not counting its height in the `y` accumulator when `render=false`) causes the height to be under-estimated.
**How to avoid:** The `y` accumulator must be incremented IDENTICALLY in both probe and render modes — the only difference is whether `ctx.fillText()` and `ctx.stroke()` are called. Test by verifying `measureAndRender(probe, skills, false) === height of final canvas`.
**Warning signs:** Last few skills' content is clipped; PNG bottom has blank space equal to one skill's worth of content.

### Pitfall 4: Regex Greedy Match Consuming Multiple Skills
**What goes wrong:** The skill regex matches from the first `<skill>` to the LAST `</skill>`, collapsing all 33 skills into one match.
**Why it happens:** A greedy `([\s\S]*)` quantifier in the skill regex matches across `</skill>` boundaries.
**How to avoid:** Use `([\s\S]*?)` (non-greedy / lazy quantifier) in the skill regex. Same for section content regex.
**Warning signs:** Only one "skill" parsed with all content merged; skill name shows only the first skill's name; separator count is 0.

### Pitfall 5: XML Section Content Indentation
**What goes wrong:** Every rendered line in the PNG has 2 leading spaces from the XML structure (because compile-knowledge.ts adds `'  ' + l` indentation to all skill element lines).
**Why it happens:** The XML compiler wraps skill elements in 2-space indentation:
```javascript
skillElements.map(el => el.split('\n').map((l) => '  ' + l).join('\n'))
```
This means section content lines in the XML have 2 leading spaces that are NOT part of the original skill file content.
**How to avoid:** When extracting section content, trim 2 leading spaces from each line (or call `.trimStart()` if more conservative). Alternative: accept the 2-space indent as visual padding.
**Warning signs:** All content lines in the PNG are slightly indented to the right of where they should start; code examples are indented 2 extra spaces.

### Pitfall 6: Output Path Targeting Wrong Directory
**What goes wrong:** `skill/lit-ui-knowledge.png` is written to `scripts/skill/` instead of the monorepo root `skill/`.
**Why it happens:** The compiler script derives `SKILL_DIR` as `path.join(__dirname, '..', 'skill')` where `__dirname` is `scripts/`. The renderer must use the same path convention.
**How to avoid:** Use `path.join(__dirname, '..', 'skill', 'lit-ui-knowledge.png')` — this resolves from `scripts/` up one level to monorepo root, then into `skill/`.
**Warning signs:** PNG not found at `skill/lit-ui-knowledge.png`; no error thrown because the write succeeded but to the wrong location.

## Open Questions

1. **How to handle very long lines that exceed 1500px canvas width**
   - What we know: At 8pt JetBrains Mono, character width ≈ 4.8px (confirmed by Phase 105 monospace invariant: `measureText('i').width === measureText('W').width = 4.8`). 1500px / 4.8px ≈ 312 characters per line. Most skill content lines are well under 312 chars.
   - What's unclear: Whether any skill file lines exceed 312 chars (long code examples, URLs, table rows).
   - Recommendation: Do NOT implement line wrapping — the image is a reference artifact for AI context, not for human reading comfort. Overflowing lines will be clipped by Skia at the canvas boundary, which is acceptable. Add a note in the script's output about max line length if desired.

2. **Should the first skill get a separator header label?**
   - What we know: PNGR-05 says "between each skill section" — implies separators appear only BETWEEN skills (i.e., 32 separators for 33 skills), not before the first one.
   - What's unclear: Whether a header label for the first skill would also be useful (to maintain visual consistency — every skill has a name label).
   - Recommendation: Show a skill name label before EVERY skill (including the first), but only draw the horizontal rule BETWEEN skills (i.e., before skills 2..33). This provides visual navigation for all 33 skills without an unnecessary rule before the very first line.

3. **Precise value for SEPARATOR_HEIGHT and line height multiplier**
   - What we know: LINE_HEIGHT = 8 × 1.5 = 12px was established in Phase 105 validate-canvas.ts. At 12px per line, a 1px rule + 12px label = ~24px separator block.
   - What's unclear: Whether 1.5× line height is correct for 8pt text in Skia, or if the ascender/descender causes visual clipping at exactly 12px intervals.
   - Recommendation: Use LINE_HEIGHT = 12px (8 × 1.5) as validated in Phase 105. If text appears visually clipped, increase to 14px (8 × 1.75). The two-pass pattern means changing LINE_HEIGHT won't cause truncation — the height auto-adjusts.
